Employee Management System.

Technology:

  • Front End: Team Builder
  • Logic: Java POJO’s
  • Database: Oracle SQL DB.

Explanation:

A Java based, basic employee management system. Simplistic in nature, this was aimed towards smaller businesses who need a cost-effective solution. We chose this particular project as it utilised the technologies we had to use during second year. Teambuilder, oracle and Java were required. Contacto Contact Management System

Technology

  • Client: Windows Phone Runtime
  • Logic: C#
  • Backend: JSON Document DB/SQL Azure in a Mobile Service

Explanation:

This choice of project stemmed from our decision on learning something new. At the time of this project, my project partner and I had only learned C++ and Java. We wanted to actively learn a new language and new methodology while building this project. We thought that Contacto- as an app- would have the specific functionality and complexity to allow for grading, while also being simple enough for a newbie to the language to learn without any problems or incidents that could cause trouble to myself and my project partner. We also wanted to try out other storage options outside a local relational database- hence, we decided on a JSON document as a Database.

Functionality:

Contacto was a functional but minimalistic contact app which allowed users to opt for a third party choice rather than the built in windows choice. A key aspect that differentiates Contacto from other phonebook/contact systems was the usage of dynamic fields. Rather than stuck to set fields specified by the system, users could add their own fields to the system- unique, intimate fields. We also added some basic fields for usage- such as email, etc. Our users could also email, text and call through the windows phone’s own library. We opted for this choice over our own so that the system could fit snuggly into the ecosystem- and allows users to specify their own email clients too. The app also featured a backup (via Azure) which allowed users to store their whole set of contacts to our cloud source, then re-download to that exact phone- we used the phone’s own ID to bind to the cloud aspect to avoid needing any sort of login.

BXP Companion App(W/AllnOne LTD)

Technology:

  • Client: Windows PHone Runtime(C#), Android, iOS(Objective C)
  • Logic: C#, Android Java, Objective C
  • Backend: RESTful deployment from BXP Server, XML

Explanation:

During the summer of 2015, I'd gotten myself an internship position in AllnOne. My assignment was to develop a mobile app companion to accompany AllnOne's key product, BXP.

BXP is a software as a service, web hosted platform for Employee and Customer management software. The key aspect to this software is the modularity- developers can create new modules to offer customers. My companion app was built to mirror this to a degree- the idea is that the mobile app should facilitate smaller aspects to the system- making appointments, etc. while leaving the bigger aspect to the Web client.

Functionality:

The mobile app offers usage of the core BXP modules to clients. From reminders, to the hamster wallboard(this is a current status and monitoring wallboard for a company)- the idea is that this contains the smaller aspects.

This app does not utilise a relational database built in-app. Rather, the app utilises a RESTful based backend. The key ingenuity to this is that the developers will not have to update the app client itself- a lot of the content is held on Server and- therefore will be easily updated from that server.
